## How to Use These Baselines

### For Parity Harness Generator (Agent 104)

The `BASELINE_INDEX.md` file in each seam directory is the **required input** for the parity-harness-generator agent. It contains:
- Screenshots inventory (file paths, descriptions, URLs)
- Data snapshots inventory (file paths, row counts)
- Workflow coverage (which workflows were captured)

**Read this file first:** `catalog-management/BASELINE_INDEX.md`

### For Visual Regression Testing

Compare migrated UI against baseline screenshots:
1. Load baseline screenshot (e.g., `01_product_list.png`)
2. Capture screenshot of migrated app at same URL
3. Compare pixel-by-pixel or use image diff tools
4. Verify layout, spacing, typography, button placement

### For API Contract Testing

Use data snapshots to define expected response structures:
1. Read `product_list_snapshot.json` to see expected product list structure
2. Read `create_product_form_structure.json` to see form field names and types
3. Define OpenAPI contract matching these structures
4. Test migrated API responses against baseline data

### For Functional Parity Testing

Use user journeys to create test scenarios:
1. Read `user-journeys.md` for step-by-step workflows
2. Implement automated tests following same steps
3. Verify migrated app produces same results as legacy

## Known Limitations

1. **Interactive Workflows:**
   - Form submissions not captured (validation errors)
   - File uploads not captured (image upload workflow)
   - Multi-step wizards not fully captured

2. **Edge Cases:**
   - Pagination beyond first page
   - Concurrent operations
   - Database constraint violations
   - Invalid input scenarios

3. **Environment Dependencies:**
   - Captures depend on database state (12 products at capture time)
   - Image files in Pics/ directory not captured (file system dependent)
